\chapter{Modèle ACO et implémentation de référence}
\label{chap:modele}

\section{Description formelle du modèle}

\subsection{Représentation de l'environnement}

L'environnement est une grille cartésienne $N \times N$ ($N = 512$). Chaque cellule $s$
stocke trois valeurs :
\begin{itemize}
    \item $c(s) \in [0, 1]$ : coût de traversée (généré par plasma fractal) ;
    \item $V_1(s) \in [0, 1]$ : phéromone de type 1, guide vers la \emph{nourriture} ;
    \item $V_2(s) \in [0, 1]$ : phéromone de type 2, guide vers le \emph{nid}.
\end{itemize}

\subsection{Dynamique des phéromones}

À chaque pas de temps, lorsqu'une fourmi visite la cellule $s$, les phéromones sont mis à jour
selon :

\begin{equation}
    V_i(s) \leftarrow
    \begin{cases}
        1 & \text{si } s \text{ est la source cible du phéromone } i \\
        \alpha \cdot \max_{s' \in \mathcal{N}(s)} V_i(s')
        + (1 - \alpha) \cdot \frac{1}{4}\sum_{s' \in \mathcal{N}(s)} V_i(s')
        & \text{sinon}
    \end{cases}
    \label{eq:pheromone_update}
\end{equation}

où $\mathcal{N}(s)$ désigne les quatre voisins (haut, bas, gauche, droite) de $s$,
et $\alpha \in [0,1]$ est le \textbf{paramètre de bruit}.

Après chaque itération, tous les phéromones s'évaporent :
\begin{equation}
    V_i(s) \leftarrow \beta \cdot V_i(s), \quad \forall s
    \label{eq:evaporation}
\end{equation}
où $\beta \in ]0, 1[$ est le \textbf{coefficient d'évaporation}.

\subsection{Règle de déplacement}

À chaque sous-pas, une fourmi en position $s$ choisit sa prochaine cellule selon :
\begin{itemize}
    \item avec probabilité $\varepsilon$ (\textbf{exploration}) : cellule voisine tirée
          uniformément parmi $\mathcal{N}(s) \setminus \{\text{obstacles}\}$ ;
    \item avec probabilité $1 - \varepsilon$ (\textbf{exploitation}) : cellule $s^*$ maximisant
          $V_{i^*}(s')$, où $i^* = 1$ si la fourmi est non chargée, $i^* = 2$ si elle est chargée.
\end{itemize}

La fourmi avance tant que son budget de mouvement $b$ (initialisé à 1) n'est pas épuisé :
$b \leftarrow b - c(s_{\text{new}})$.

\section{Génération du terrain fractal}

Le terrain est généré par l'algorithme \textit{midpoint displacement} récursif :
une grille de $n \times n$ sous-grilles de taille $2^k + 1$ est initialisée aux coins,
puis subdivisée récursivement en calculant les points médians avec une déviation bornée par
$d \times \frac{ns}{2}$. Les valeurs sont ensuite normalisées dans $[0, 1]$.

\section{Architecture du code de référence}

Le code est organisé en quatre modules :

\begin{description}
    \item[\texttt{fractal\_land}] Génération et stockage du terrain (tableau 1D de \texttt{double}).
    \item[\texttt{pheronome}] Grille $(N+2)^2$ de paires $[V_1, V_2]$, avec cellules fantômes
          à $-1$ pour les bords. Méthodes : \texttt{mark\_pheronome()}, \texttt{do\_evaporation()},
          \texttt{update()} (double-buffering).
    \item[\texttt{ant}] Un objet par fourmi (\textbf{AoS}) : état, position (\texttt{SDL\_Point}),
          graine aléatoire. Méthode \texttt{advance()} : boucle de sous-pas jusqu'à budget épuisé.
    \item[\texttt{ant\_simu}] Point d'entrée, boucle principale SDL, appel à
          \texttt{advance\_time()} à chaque itération.
\end{description}

\subsection{Analyse de complexité}

La complexité par itération est dominée par la phase de déplacement des fourmis :
$\mathcal{O}(m \cdot \bar{k})$ où $\bar{k}$ est le nombre moyen de sous-pas par fourmi
(dépend du terrain et de la position). La phase d'évaporation est $\mathcal{O}(N^2)$, fixe
et régulière.

\subsection{Structures de données : AoS et SoA}
% ..............................................................

Lorsqu'un programme manipule une population d'agents (ici des fourmis), il doit
choisir comment organiser leurs données en mémoire. Deux paradigmes s'opposent :

\paragraph{AoS — \textit{Array of Structures} (Tableau de structures).}
Chaque agent est représenté par un objet regroupant tous ses attributs :

\begin{lstlisting}[language=C++, caption={Organisation AoS — un objet par fourmi}]
struct ant {
    state_t    state;     // etat (chargee / non chargee)
    position_t position; // coordonnees (x, y)
    size_t     seed;      // graine aleatoire
};
std::vector<ant> colony(m);  // m fourmis contigues par objet
\end{lstlisting}

Les attributs d'une fourmi sont contigus en mémoire, mais un parcours de tous les
\texttt{state} implique des sauts de \texttt{sizeof(ant)} octets, dégradant la
prélecture matérielle (\textit{hardware prefetcher}) et l'utilisation du cache.

\paragraph{SoA — \textit{Structure of Arrays} (Structure de tableaux).}
Les mêmes agents sont représentés par \emph{un tableau par attribut} :

\begin{lstlisting}[language=C++, caption={Organisation SoA — un tableau par attribut}]
class ant_colony {
    std::vector<state_t>    m_states;    // tous les etats
    std::vector<position_t> m_positions; // toutes les positions
    std::vector<size_t>     m_seeds;     // toutes les graines
};
\end{lstlisting}

L'accès à un attribut particulier pour toutes les fourmis devient linéaire en mémoire,
ce qui maximise le remplissage des lignes de cache et favorise la vectorisation
automatique par le compilateur (instructions SIMD : SSE4, AVX2~\cite{intel2012avx}).
Cet avantage est particulièrement net dans les boucles multi-thread, où la répartition
d'un tableau contigu entre cœurs minimise les conflits de cache.

\subsection{Parallélisation en mémoire partagée : OpenMP}
% ..............................................................

\textbf{OpenMP} (\textit{Open Multi-Processing}) est une interface de programmation
standardisée pour la parallélisation en \textbf{mémoire partagée}, supportée
nativement par GCC, Clang et MSVC~\cite{openmp5}. Son modèle repose sur le
paradigme \textit{fork-join} : un thread maître crée dynamiquement une équipe de
threads à chaque région parallèle, puis les rejoint à la sortie.

La directive centrale est \texttt{\#pragma omp parallel for}, qui distribue
les itérations d'une boucle entre les threads disponibles :

\begin{lstlisting}[language=C++, caption={Parallélisation d'une boucle avec OpenMP}]
#pragma omp parallel for schedule(dynamic, 16) \
                         reduction(+:local_food)
for (size_t i = 0; i < m; ++i)
    advance_one(i, ...);  // chaque fourmi traitee par un thread
\end{lstlisting}

Les clauses principales utilisées dans ce projet sont :
\begin{itemize}
    \item \textbf{\texttt{schedule(dynamic, chunk)}} : distribue les itérations par
          paquets de \texttt{chunk} de façon dynamique, adapté aux charges irrégulières
          (le coût d'avancement d'une fourmi dépend du terrain) ;
    \item \textbf{\texttt{reduction(+:var)}} : chaque thread accumule une valeur locale,
          puis les contributions sont sommées atomiquement à la fin de la région parallèle,
          évitant toute course critique sur \texttt{cpteur\_food}.
\end{itemize}

L'efficacité d'OpenMP est bornée par la \textbf{loi d'Amdahl}~\cite{amdahl1967validity} :
si $f$ est la fraction parallélisable du code, l'accélération maximale tend vers
$1/(1-f)$ quel que soit le nombre de threads $p$.

\subsection{Parallélisation en mémoire distribuée : MPI}
% ..............................................................

\textbf{MPI} (\textit{Message Passing Interface}) est le standard de facto pour la
programmation parallèle en \textbf{mémoire distribuée}~\cite{mpi31}. Chaque processus
dispose de son propre espace d'adressage privé ; les données sont échangées
explicitement via des messages (\texttt{MPI\_Send}, \texttt{MPI\_Recv}) ou des
opérations collectives synchronisées sur tous les processus d'un communicateur.

L'opération collective clé dans ce projet est \textbf{\texttt{MPI\_Allreduce}} :
chaque processus contribue un buffer local, et tous reçoivent le résultat de la
réduction globale (ici le maximum cellule par cellule des phéromones).

\begin{lstlisting}[language=C++, caption={Synchronisation globale des phéromones via MPI}]
// Chaque processus a mis a jour son buffer local de pheromones.
// MPI_Allreduce fusionne les buffers en prenant le max par cellule,
// et distribue le resultat a tous les processus.
MPI_Allreduce(MPI_IN_PLACE, buf, count,
              MPI_DOUBLE, MPI_MAX, MPI_COMM_WORLD);
\end{lstlisting}

Contrairement à OpenMP, MPI implique un \textbf{coût de communication} proportionnel
au volume de données échangé. Le rapport calcul/communication détermine si la
parallélisation MPI est rentable : si les communications dominent, l'ajout de
processus peut dégrader les performances~\cite{gropp1999mpi}. Ce phénomène est
central dans l'analyse de l'Approche 1 MPI (chapitre~\ref{chap:mpi}).

La loi de \textbf{Gustafson}~\cite{gustafson1988reevaluating} complète Amdahl
pour les problèmes à taille variable : en augmentant la taille du problème avec
le nombre de processus, l'efficacité peut rester élevée même pour une fraction
séquentielle non nulle.

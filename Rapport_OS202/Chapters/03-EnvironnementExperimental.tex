\chapter{Environnement expérimental}
\label{chap:env}

Les expériences ont été conduites sur \textbf{deux machines distinctes} afin
d'évaluer si les tendances de performance observées — profils de speedup, saturation,
goulots d'étranglement — sont spécifiques à une architecture ou généralisables.

\subsection{Architecture mémoire (lstopo)}

La figure~\ref{fig:lstopo_loic} illustre la hiérarchie mémoire telle que rapportée
par l'outil \texttt{hwloc/lstopo}~\cite{broquedis2010hwloc}. On distingue clairement :

\begin{itemize}
    \item les \textbf{6 P-cores} (cœurs de performance, à gauche), chacun avec un
          cache L2 privé de 1 280 Ko et l'Hyper-Threading (2 threads logiques/cœur) ;
    \item les \textbf{8 E-cores} (cœurs d'efficacité, à droite), regroupés en
          deux clusters de 4, partageant chacun un cache L2 de 2 048 Ko, sans HT ;
    \item le \textbf{cache L3 de 24 Mo} partagé entre tous les cœurs ;
    \item le nœud NUMA unique (15 Go de RAM visible par le système).
\end{itemize}

\subsection{Implications pour la parallélisation}

L'architecture hybride introduit une \textbf{asymétrie de performance} entre threads.
Les 12 threads des P-cores (6 cœurs × 2 HT) offrent la meilleure latence
et la bande passante L2 la plus large. Les 8 threads des E-cores sont plus lents
(fréquence et largeur de pipeline réduits) mais économes en énergie. Lors des
benchmarks OpenMP, cette asymétrie se traduit par une saturation des gains au-delà
de 12 threads, un plateau caractéristique de ce type de processeur hybride.

\subsection{Paramètres de simulation fixes}

Pour assurer la comparabilité des mesures entre les deux machines, les paramètres
de la simulation sont maintenus constants (voir aussi le chapitre~\ref{chap:modele}) :

\begin{itemize}
    \item Grille $N = 512$ cellules par côté ;
    \item $m = 5\,000$ fourmis ;
    \item Paramètres : $\varepsilon = 0.8$, $\alpha = 0.7$, $\beta = 0.999$ ;
    \item Graine aléatoire initiale : \texttt{2026} (reproductibilité).
\end{itemize}

\subsection{Procédure de collecte}

Chaque exécution instrumentée écrit ses données temporelles dans un fichier
\texttt{profiling.csv} (une ligne par itération), selon le format :

\begin{lstlisting}[language={},caption={Format du fichier profiling.csv}]
iteration,t_ants_ms,t_evap_ms,t_update_ms,t_render_ms,food_count
0,4.1521,0.6489,0.0118,97.3214,0
1,4.2013,0.6601,0.0121,98.1042,0
...
\end{lstlisting}

Un programme d'analyse \texttt{profile\_stats.exe} agrège ensuite ces données et
calcule pour chaque phase : \textbf{moyenne}, \textbf{médiane}, \textbf{min},
\textbf{max} et \textbf{écart-type}. Les tableaux de résultats présentés dans les
chapitres suivants sont issus de cette agrégation.

\subsection{Métriques de scalabilité}

Pour comparer les versions parallèles, on utilise les métriques classiques du
calcul haute performance~\cite{gropp1999mpi} :

\begin{equation}
    S(p) = \frac{T_1}{T_p}
    \qquad \text{(accélération)}
    \label{eq:speedup}
\end{equation}

\begin{equation}
    E(p) = \frac{S(p)}{p}
    \qquad \text{(efficacité parallèle)}
    \label{eq:efficiency}
\end{equation}

où $T_1$ est le temps de la version séquentielle de référence et $T_p$ le temps
avec $p$ unités de calcul. L'efficacité idéale est $E(p) = 1$ (100\,\%).

\subsection{Reproductibilité}

\paragraph{Reproductibilité fonctionnelle.}
Avec les mêmes paramètres et la même graine, les indicateurs métier (première
nourriture ramenée au nid, quantité totale collectée) doivent rester stables
entre runs sur la même machine.

\paragraph{Reproductibilité des performances.}
Les temps absolus (ms/it) ne sont \emph{pas} transférables d'une machine à l'autre
(fréquence CPU, hiérarchie mémoire, politique d'ordonnancement du système). En
revanche, les \textbf{parts relatives par phase} et les \textbf{tendances de speedup}
doivent être qualitativement similaires si les architectures sont comparables.

\paragraph{Version OpenMP.}
La race condition délibérée sur \texttt{mark\_pheronome()} rend le comportement
fonctionnel non strictement déterministe en mode multi-thread. Les indicateurs
métier peuvent varier de $\pm 5\,\%$ entre runs OpenMP, ce qui est acceptable
pour une évaluation de scalabilité.
